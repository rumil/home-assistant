# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A **Home Assistant blueprint-publishing repo**, not an HA config directory. It tracks a single automation blueprint, `blueprints/switch-light-sync.yaml`. The live HA instance does not read this repo directly — it imports the blueprint from the file's `source_url` (the raw GitHub URL on `main`). There is no application to build, and no test/lint suite; the deliverable is declarative YAML validated by Home Assistant at import/instantiation time.

## Deploy loop (how a change reaches live HA)

1. Edit `blueprints/switch-light-sync.yaml`, commit, push to `main`.
2. In the HA UI: **Settings → Automations & Scenes → Blueprints →** the blueprint **→ ⋮ → Re-import blueprint**.

Hard constraints (do not rediscover the hard way):
- The Home Assistant MCP `ha_import_blueprint` tool **cannot overwrite** an existing blueprint, and there is no blueprint delete/override tool. The re-import is therefore a **manual UI step** — it cannot be automated from here.
- GitHub's raw CDN caches ~5 minutes; if a re-import doesn't reflect the latest push, wait and re-import again.
- After re-importing a changed blueprint, existing automation instances that use it must be re-validated. Adding a **required** input breaks every existing instance until each is updated — prefer adding new inputs with a `default:` so re-import stays non-breaking.

## Validating changes

- `python3` here has no `yaml` module. For a quick local syntax check use Ruby (it must tolerate the `!input` tag):
  ```bash
  ruby -ryaml -e 'YAML.add_domain_type("","input"){|t,v| {"input"=>v}}; YAML.load_file("blueprints/switch-light-sync.yaml"); puts "OK"'
  ```
- Real validation is HA-side. Use the Home Assistant MCP: `ha_eval_template` to test Jinja snippets against live entity state before embedding them, and `ha_get_blueprint` / `ha_config_get_automation` to confirm the live blueprint and instances after a re-import.

## Debugging a live misfire

**Start with the trace — it now states the decision outright.** The pipeline computes its classification into `variables:`, so a trace shows `intent`, `is_self_boot`, `is_recovery`, `needs_wake`, `device_originated`, `at_boot_level`, `from_on`, `target_state` and `apply_data` directly. There is no need to re-derive which path ran from timings or the recorder; read the variables block of the first `action/0` step.

The five live instances set `stored_traces: 30` (HA's default is 5), so roughly the last 30 runs are retained per automation — enough for a burst plus the events around it, but a busy morning still rotates an incident out within the hour. Once traces are gone, reconstruct from the recorder:

- `ha_get_history` on the light, the switch, **and** the instance's `input_text` store, `order="asc"`. The store's transitions show what each run decided to persist.
- `ha_get_logs(source="logbook", compact=False)` on the switch/light — `context_user_id` distinguishes a **user action** from a **device report** (firmware boot: no context at all) from an **automation's own call** (`context_domain: automation`).
- Timing fingerprints the branch when the trace is gone: an observe run is **~1.0 s**; a restore run with priming is **~11 s** (1 s settle + prime 1 s + 4 s watch + restore 1 s + 4 s watch); Scenario A is **~1.0 s**. A light-side run lasting more than a couple of seconds means `intent: restore`.
- `ha_get_logs(source="system")` — a burst of `<name>: Already running` entries means triggers are being dropped; cross-check against the desync window rather than assuming they are benign. The blueprint sets `max_exceeded: debug`, so these **no longer appear at WARNING** — an outage or restart makes `mode: single` drop dozens of self-inflicted re-entrant triggers per instance, which is by design and drowned the log. To see them, raise the level first: `logger.set_level` on `homeassistant.components.automation.<object_id>` = `debug`, then read with `level="DEBUG"`.
- Recorder/logbook timestamps come back in **local time** (Europe/Warsaw) while trace timestamps are **UTC** — do not compare them without converting.

## Blueprint architecture (`switch-light-sync.yaml`)

Two-way sync between a **Grenton virtual switch** and a **light** entity, resilient to restarts and power outages. `mode: single` — this is load-bearing: it drops the re-entrant triggers caused by the blueprint's own service calls, so scenarios can act on the light without fighting themselves.

`mode: single` cuts both ways, and the cost is not theoretical. A running scenario holds the only slot, so **any** state change landing mid-run is dropped (`execution: failed_single` in the trace) — including one that was needed to correct the state. There is no periodic reconcile, so a desync created this way persists until the next outage or user action. This makes the settle invariant below a correctness requirement, not a nicety: a scenario that reads stale state and drives an entity wrong has no second chance to fix it.

**Triggers** (each tagged with an `id`, dispatched by a top-level `choose`):
- `grenton_changed` / `grenton_recovered` — Grenton switch real change vs. return from unknown/unavailable.
- `light_changed` — light on↔off (real→real). Deliberately **not** debounced: the on/off sync must stay immediate.
- `light_recovered` — light returns from unknown/unavailable (outage recovery).
- `light_low` — `attribute: brightness` trigger with **`for: "00:00:03"`**; catches a firmware boot that drops to ~1% **without** any state-string change (these lights sometimes report 1% without going unavailable). State triggers do not fire on attribute-only changes, hence the dedicated trigger. The debounce discards the turn-on ramp (see the ramp gotcha); a genuine boot sits at 1% indefinitely, so it still fires, 3 s later.
- `ha_start` — HA finished starting.

**Three branches** (top-level `choose` — **order is load-bearing**, first match wins):

- **A (`grenton_changed`)** — user action from the Grenton side. Drive the light to match the switch, settle, save. **Single-shot on purpose:** see the re-assert gotcha. It acts before it settles, because a settle up front would add visible lag to every wall-switch press — that is the only reason it does not share the pipeline below.

- **The light-side pipeline (`light_changed | light_low | light_recovered | ha_start`, when `restore_previous` and `has_store`)** — recovery, self-boot and ordinary user actions are the same physical situation seen at different moments, so they share one implementation with a single branch point:

  ```
  settle (delay 1s + wait for brightness)
  classify → intent = restore | observe
  target  := snapshot (restore) | current light state (observe)
  prime    (restore, threshold > 0)
  assert   (skipped when there is nothing to assert)
  sync switch
  save     (observe only)
  ```

  `intent == 'restore'` writes the **light from the store**; `intent == 'observe'` writes the **store from the light**. They are inverse data flows, which is why both must exist: drop the save and nothing ever learns what "previous" means; always restore and you can never switch a light off.

  Classification happens *inside* the branch, not in `conditions:` — the boot test needs a brightness reading, and the state string arrives ~1.5 s before the attribute, so a condition would read the value before it exists.

- **Catch-all (`grenton_recovered | light_recovered | ha_start`)** — must stay **last**. The light is source of truth; sync the switch to it. Takes Grenton's own recovery (handled nowhere else, relevant to every instance) plus everything the pipeline declined: `light_source_of_truth` instances, and `restore_previous` instances with no store configured.

**Intent classification** (all visible in the trace):
- `is_recovery` — trigger was `light_recovered` or `ha_start`.
- `is_self_boot` — the bulb powered itself up: went **off → on**, at a **readable** brightness ≤ threshold, and `device_originated`. The light never went unavailable so no recovery trigger fired, but physically it is the same event and gets the same treatment.
- `device_originated` — `trigger.to_state.context` has **both** `user_id` and `parent_id` null. Both halves matter: a Grenton wall-switch press reaches the light through Scenario A, which leaves `user_id` null but **does** stamp `parent_id`, so `user_id` alone would take it for a device report.
- `needs_wake` — an observe-path boot-level drop (the light was already on and sank). Corrects **brightness only**, never on/off.

**Persistence model:** the "saved state" is a JSON snapshot written to a per-instance `input_text` helper (`state_store` input) via `input_text.set_value`, and read back with `from_json`. This is deliberate: an HA automation **cannot** create a persistent scene (`scene.create` is wiped on restart) or create helpers at runtime, so the store helper is provisioned separately and passed in as an input. `saved_attributes` (input) selects which light attributes to snapshot; the save loop skips any the light reports as `none`. Saves never persist a non-on/off state, an unreadable brightness, or a low brightness that is not a deliberate dim (see the ramp gotcha).

**`post_outage_behavior`** has only two values: `light_source_of_truth` (default) and `restore_previous`. `turn_on`/`turn_off` were removed in `27eb974` — unused by all five live instances. The pipeline is gated on `restore_previous and has_store`, **not** on the threshold, so `recovery_brightness_threshold: 0` disables boot detection and priming only; the snapshot is still restored on recovery.

## Blueprint conventions / gotchas

- **There are two distinct ways these lights turn themselves off. Do not conflate them — they have opposite culprits.** Diagnosed 2026-07-20/21 over a long investigation that went down the wrong path twice.
  - **(A) Spurious device dropout — the blueprint is innocent.** ~3 s after the first turn-on following a Zigbee re-establishment, *every* group member reports `state: OFF` **at full brightness** (255, not the 1 % boot level) and Z2M computes the group as off. In the traces `at_boot_level`/`is_self_boot`/`needs_wake` are **false**, `cur_bri` is **255**, no `light.turn_off` appears anywhere, and the HA `off` state carries `parent_id: null` (an automation call always stamps a parent). Two further tells: it only bites when several groups are switched **in a rapid burst** (the first light switched always survives; the 2nd/3rd/4th drop), and **the fragility does not expire** — it persists for as long as the light stays off, so a 00:14 recovery still ambushed the first turn-on at 11:26. This is what the pipeline's **priming loop** exists to absorb, and on 2026-07-22 it absorbed a four-instance simultaneous recovery burst — the exact provoking load — without consuming a single retry.
  - **(B) The boot heuristic hijacking a user turn-on — the blueprint's fault.** If the bulb genuinely comes up at 1 % (`cur_bri: 1`, common right after a Z2M/HA restart) the heuristic fires and "restores" the snapshot; when that snapshot is `{"state":"off"}` it **switches off a light the user just turned on**. Caught red-handed in a trace on 2026-07-21.
  - This was once believed unfixable by tightening the user test, on the theory that a post-restart device report carries no user context even when the user did initiate the turn-on. **That theory is wrong** — disproven 2026-07-22 across many traces, where `device_originated` came out **false** on every genuine user action, including turn-ons moments after a Z2M restart. What was actually missing was `parent_id`: `user_id` alone cannot see a Grenton wall-switch press, which arrives through Scenario A with `user_id` null and a parent stamped. Requiring **both** to be null is a sound test, and it is what gates the self-boot restore today. The old structural rule still holds on the observe path — `needs_wake` corrects brightness only — but on/off restores are no longer confined to the unavailable-recovery trigger.
- **Priming the bulb and making the restore stick are two separate problems.** The pipeline has **two** retry loops and needs both. The first proves the bulb accepts an ON; the second re-asserts the *restored* target state until it holds. Verifying only the first is not enough: on 2026-07-21 office primed cleanly, took a correct `turn_off`, and then flipped back on **3.7 s later** because the group members disagreed (`linear_1` reported `OFF` while the spots reported `ON`, and Z2M computes a group as on when **any** member is on). Staying inside the run for that second loop is half the fix — it holds the `mode: single` slot so the spurious flip's `light_changed` is *dropped*; once the run had ended, the user-action path took the flip for a user action, synced the switch on and wrote `{"state":"on"}` into the store.
- **Never re-assert against a change HA itself caused — a loop that cannot tell noise from intent is not a correction loop, it is a fight.** Learned expensively on 2026-07-22 (`6684f88`): a re-assert loop added to Scenario A "for consistency" switched off a light the user had just turned on, 3 s after they turned it on (trace `0ce61722`: iteration 1 commanded nothing because the light already matched, the drift watch then *completed* when the user turned it on, and iteration 2 dutifully "corrected" it). It fired constantly because A runs on every Grenton change **including the switch sync the pipeline itself performs**, so routine activity parked the automation in a 4 s watch window with the `mode: single` slot held — which also swallowed the user's own `light_changed`. Two consequences, both now in the code: **Scenario A is single-shot** (it is driven by the switch and has no signal for what caused a change to the light), and the pipeline's assert loop stops on a foreign change —

  | origin | the light's context | re-assert? |
  |---|---|---|
  | our own command | this run's `context.id` (HA reuses it for the device echo, 5 s) | yes |
  | spurious device flip | fresh, no user, no parent | yes |
  | user action | `user_id` set | **stop** |
  | another automation | a parent that is not ours | **stop** |
- **The turn-on ramp is not one event.** Z2M publishes a turn-on as a sequence of brightness messages — `1 → 252 → 255` over ~10 s measured on these groups — each one its own `light_low` trigger. Before `b55405a` a single turn-on produced four runs and wrote `{"brightness":1}` then `252` then `255` into the store. The low value survived the save guard because the guard only asked "did a user cause this", the carve-out for a deliberate dim to 1 %. **You cannot dim a light you just turned on** — a real dim requires `from_on`, so a low reading on an off→on transition is always the ramp caught mid-flight, whoever initiated it. Persist a brightness ≤ threshold only when `from_on and not device_originated`. Usually the next run overwrote the bad value, so this only bit if a turn-on was the last event before an outage — the store would hold `1` and the next recovery would restore the room to 1 %.
- **A poisoned store is silent and self-perpetuating.** It only shows up on the *next* recovery, as the light being restored to the wrong state. When a recovery ends wrong, check `input_text.light_sync_*` before re-deriving anything.
- **Never let the priming level reach the snapshot.** The save is gated on `intent == 'observe'` for exactly this reason, and the re-entrant triggers priming generates are dropped by `mode: single` while the run holds the slot. If you let a save reach the restore path, the priming level becomes the "previous" state and the next recovery restores the light to it.
- **`prime_brightness` must stay above `recovery_brightness_threshold`.** The priming loop's "did it drop out?" test treats *at or below threshold* as a dropout, so priming at or under the threshold makes the loop think it failed every time and burn all its retries.
- **The priming level is also the level the bulbs *remember*.** These bulbs retain the last level they were commanded to and light up at it on a plain `light.turn_on` with no `brightness`. So after a recovery that ends with the light off, the last thing they were told was the priming level — and that is what the next manual turn-on lights up at. Priming at 10 % (the original default of 25) left every fixture coming on dim; hence the default is now **255**. If you ever lower it to make the recovery flash subtler, expect that dimness to resurface on the next plain turn-on.

- **`context.user_id` inside an automation is always `none` — it can never detect a user action.** HA builds an automation's run context as `Context(parent_id=<trigger context id>)` and does **not** copy `user_id` into it. Any `context.user_id is none` test is therefore unconditionally true and is dead code. Use **`trigger.to_state.context.user_id`**, which does carry the real user: HA's entity service call stamps the calling context onto the entity, and subsequent `async_write_ha_state()` calls reuse it for `CONTEXT_RECENT_TIME` (5 s) — long enough that even the *later* MQTT state/attribute messages from Z2M still carry it. This is why the logbook and the trace disagree: the logbook resolves attribution by walking the parent chain and shows the real `context_user_id`, while the automation reads the raw value and sees `null`. Do not diagnose this from the logbook alone. Cost of getting it wrong: the boot heuristic in the old Scenario B fired on deliberate user turn-ons and "restored" the light to the saved snapshot (`{"state":"off"}`) ~3.5 s later, across all four instances (fixed 2026-07-16). Note `trigger.to_state.context` is also where **`parent_id`** lives, which is the half that distinguishes an automation-driven change from a device report — see the intent classification above.
- **A light attribute that is absent reads as `none`, and `| int(0)` turns that into a real `0`.** HA reports *every* light attribute as `none` while the entity is off, and for a moment after it turns on. So `state_attr(light, 'brightness') | int(0) <= threshold` is **true for an unreadable brightness**, and any "is it dimmed to ~1%?" test silently becomes "…or did we fail to read it?". Always require `state_attr(...) is not none` alongside the threshold compare. Diagnostic that proves it in one shot: `ha_eval_template` the old and new forms side by side against an off light — old yields `True`, new `False`.
- **Waiting for the state string is not waiting for the attributes.** Z2M publishes a group's `state` and its `brightness` in separate MQTT messages; the gap was measured at **1.53 s** on these groups, so the old fixed `delay: "00:00:01"` sampled while brightness was still absent. Wait for the reading itself (`wait_template` on `state_attr(...) is not none`, guarded by `not is_state(..., 'on')` so an off light passes straight through). Keep the fixed settle **as well** — the two solve different problems: the settle guards against reading an *intermediate* value the light is about to leave, the wait guards against reading *no* value at all. Collapsing them into one re-opens the other hole.
- **Settle after every service call before reading the state it changes.** `light.turn_on`/`turn_off` return as soon as the command is dispatched; the entity keeps reporting its *old* state until the device round-trips (MQTT/Zigbee: ~100–500 ms). A `condition: state` evaluated straight after a light command reads the pre-command value. Every restore→sync path therefore settles between the command and the read; the old Scenario B's boot path silently lacked it until 2026-07-16 (fixed in `836978b`), which drove the Grenton switch to the state the light was *leaving* and — because `mode: single` then dropped the correcting `light_changed` — stranded a switch `on` against a light `off` for 2h10m. **If you add a branch that commands an entity and then reads it, settle in between.** The same reasoning applies to the brightness read the pipeline's classification depends on (the 1 s at the top) — Zigbee groups report `state: on` first and the `brightness` attribute ~500 ms later, so reading immediately sees `brightness: 0` and can misfire the boot test. Since `27eb974` there is **one** settle definition and **one** boot-level definition shared by every path; before that there were three of each, honoured inconsistently (`prime_brightness` was respected on one path out of three, and the single-shot restore in the old Scenario D was a latent repeat of the 2026-07-21 flip-back).
- Target lights are **Zigbee2MQTT groups** (`platform: mqtt`, `unique_id: <n>_light_zigbee2mqtt`) — Z2M publishes the group as its own MQTT light and computes group state itself. The `group_entities` attribute is Z2M metadata, not an HA group-helper member list, so those member entity_ids may not resolve in HA (they are deliberately disabled on the live instance) and carry no recorder history. This does not affect the group's behaviour.
- **Read the Zigbee2MQTT add-on log — it is the per-fixture trail HA does not have, and it shows values HA actively hides.** `ha_get_logs(source="supervisor", slug="45df7312_zigbee2mqtt", search="<device_or_group_topic>")`. It carries every member's raw payload (`brightness`, `color_mode`, `color_temp`, `linkquality`, `state`) plus the computed group topic. Reach for it **early**, alongside `ha_get_history` / `ha_get_automation_traces`, on any Zigbee issue — the recorder cannot substitute: `ha_get_history` elides most attributes, and HA nulls *every* light attribute while the entity is off, so a device's real reported values are visible nowhere else. Two 2026-07-16 diagnoses turned on it: the per-device `brightness` proved a "1% boot" reading was actually an unread `brightness: None` (the office bulbs were demonstrably at 254 yet still took the boot path — impossible unless `cur_bri` read 0), and the per-device `color_mode: "hs"` explained why a group's `color_temp_kelvin` was permanently `none` (one `hs` member poisons the group's computed mode, which HA then rejects against `supported_color_modes: ["color_temp"]` — look for `Invalid color mode 'hs' received` at WARNING). It is also the fastest way to confirm a fix at the device level rather than through HA's interpretation of it.
- **Optional `entity` inputs must be referenced via a template, not `!input`.** An unset entity input defaults to `""`; if `!input` puts that empty string into a service `target: entity_id:`, HA fails config-load validation for *every* instance (checked at load, regardless of runtime guards). Use `entity_id: "{{ some_var }}"` and gate the call on a `has_store`-style guard instead. See `state_store` for the pattern.
- Bind inputs to `variables:` and use the variable inside Jinja — `!input` is a YAML tag, not a template value.
- Light color attributes use `color_temp_kelvin` (mireds-based `color_temp` was removed in HA 2026.3).
- Keep `source_url` stable — it is the re-import anchor.
